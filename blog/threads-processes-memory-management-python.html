Threads, Processes, and Memory Management in Python

Today’s notes delve into the inner workings of Python's execution model, focusing on how processes and threads help manage tasks efficiently. As an engineer working on systems that require both CPU-intensive computation and I/O operations, understanding these concepts is crucial to balancing performance and isolation.

Process - an independent running program program with it's own isolated memory and resources.

A process contains:
* virtual memory space
* program stack
* heap
* threads
* file handles
* OS-managed resources
Processes are isolated from one another by the operating system.

Examples)

  • Chrome browser process
  • Python application
  • Database server process

Processes communicate through:

  • sockets
  • pipes
  • shared memory
  • IPC mechanisms

Tradeoff:

  • safer isolation
  • heavier than threads

Thread - a lightweight prcoess, shared everything with parent process that spawned it up. Only things that absolutely require, technically access to entirev virtual memory of parent process.

Threads within the same process share:

  • virtual memory
  • heap
  • file handles
  • global variables

Each thread has its own:

  • stack
  • registers
  • instruction pointer

Threads are useful when tasks need shared memory access.

Examples)

  • web server request handling
  • background workers
  • UI thread + worker thread
  • concurrent API calls

Tradeoff:

  • cheaper than processes
  • shared state introduces synchronization complexity

Failure modes:

  • race conditions
  • deadlocks
  • data corruption

When to use:

  • shared in-memory state
  • I/O concurrency
  • lightweight parallelism
  • async task coordination

When NOT to use:

  • CPU-heavy Python workloads under the GIL
  • strict isolation requirements

Program Stack - memory region storing function call state and local variables.

The stack contains:

  • local variables
  • function arguments
  • return addresses
  • stack frames

Characteristics:

  • automatically managed
  • very fast
  • typically smaller than heap
  • grows/shrinks with function calls

Example)

def add(a, b):
    result = a + b
    return result
# Variables like:
a
b
result

typically live on the stack.

Failure mode:

  • stack overflow

Program Heap - memory region used for dynamically allocated memory.

Heap memory is used for:

  • objects
  • large data structures
  • dynamic allocation

Characteristics:

  • larger than stack
  • slower allocation
  • manually managed in some languages
  • garbage collected in others

Examples)

  • Python objects
  • Java objects
  • dynamically allocated arrays

Heap memory persists beyond function scope until released.

Virtual Memory - abstraction layer allowing processes to believe they own contiguous memory.

The operating system maps:

virtual addresses
↓
physical memory pages

Benefits:

  • process isolation
  • memory protection
  • larger apparent memory space
  • simplified programming model

Virtual memory allows:

  • paging
  • swapping
  • memory-mapped files

memory-mapped files - files mapped directly into a process’s virtual memory space.

Instead of:

file.read()

the OS maps the file into memory.

File on Disk
↓
Virtual Memory Mapping
↓
Access like normal memory

Benefits:

  • Fast file access
  • OS manages caching
  • Efficient for large files
  • Multiple processes can share mappings

Common uses:

  • databases
  • search indexes
  • large datasets
  • machine learning datasets

Examples:

  • SQLite
  • LMDB
  • Lucene
  • vector databases

Page - fixed-size block of memory used by virtual memory systems.

Memory is divided into pages.

Typical size: 4 KB

The OS maps:

virtual pages
↓
physical pages

Paging enables:

  • memory isolation
  • swapping
  • efficient memory management

Important concepts:

  • page faults
  • page tables
  • swapping
  • TLB (Translation Lookaside Buffer)

Performance issue:

  • excessive page faults can severely slow systems

Virtual Pages vs Physical Pages

Modern systems use virtual memory.

Each process sees: Virtual Address Space
while actual RAM contains: Physical Memory

The OS maps: Virutal Pages -> Physical Pages

Process Believes:
Address 0x1000
Address 0x2000
Address 0x3000

exist continuously, but internally:

Virtual Page A → Physical Page 57
Virtual Page B → Physical Page 103
Virtual Page C → Physical Page 12

Benefits:

  • process isolation
  • memory protection
  • simplified programming model
  • memory sharing

Kernel - core component of the operating system.

The kernel manages:
- memory
- processes
- threads
- hardware access
- scheduling
- filesystems
- networking

The kernel is the intermediary between applications <-> hardware.

i.e. Linux kernel, XNU kernel (macOS), Windows NT kernel

Synchronous - operations execute in a sequence and wait for completion before continuing.

data = fetch_data() # execution blocks until process finishes
process(data)

Characteristics:
- simpler reasoning
- easier debugging
- blocking behavior

Good for:
- simple workflows
- deterministic execution
- low concurrency requirements

Asynchronous - operations can make progress without blocking the current execution flow.

data = await fetch_data()

while waiting:
- other tasks can execute
- event loop continues
- resources are utilized efficiently

Good for:
- network I/O
- APIs
- databases
- high-concurrency servers

Important concepts:
- event loop
- future/promises
- coroutines
- callbacks

Async improves:
- concurrency
- throughput

Async does NOT necessarily improve:
- CPU-bound computation

Multithreading - multiple threads executing within the same process.

Threads share:
- heap
- virtual memory
- process resources

Benefits:

  • lightweight concurrency
  • shared memory access
  • lower overhead than processes

Common use cases:

  • web servers
  • background tasks
  • UI responsiveness
  • concurrent I/O

Tradeoffs:

  • synchronization complexity
  • race conditions
  • deadlocks

Python note:

  • CPython has the GIL (Global Interpreter Lock)
  • CPU-heavy Python threads may not execute truly in parallel

Multiprocessing - multiple independent processes executing concurrently.

Each process has:

  • separate memory space
  • separate virtual memory
  • separate heap

Benefits:

  • true parallelism
  • stronger isolation
  • bypasses Python GIL

Tradeoffs:

  • heavier memory usage
  • IPC overhead
  • process startup cost

Good for:

  • CPU-heavy workloads
  • ML preprocessing
  • distributed computation
  • parallel workers

Examples)

  • worker pools
  • distributed systems
  • data pipelines

Reference Counting (Most Common)

Every Python object keeps a count of how many references point to it.
Reference count = 0.

At that moment CPython immediately frees the object.

Garbage Collector (Cycles)

Reference counting cannot handle cycles.

the two objects still reference each other.

Reference count never reaches 0.

Python’s cyclic garbage collector detects this and cleans it up later.

Function Scope Cleanup

When a function returns:
data goes out of scope.

Reference count drops to 0.

Memory becomes eligible for release.

Explicit Deletion

You can remove a reference:

del my_list

This does not directly free memory.

It removes the name: my_list
from the current scope.

If no references remain:

then the object is freed.

Swap - disk space used as overflow memory when RAM becomes full.

The operating system can move inactive memory pages from RAM to disk.

RAM Full
↓
Move inactive pages
↓
Swap Space (Disk)

Benefits:

  • Allows more memory than physical RAM
  • Prevents immediate process termination

Tradeoffs:

  • Disk is much slower than RAM
  • Excessive swapping causes severe performance degradation

Thrashing - Excessive swapping causes severe performance degradation

Occurs when:

Page In
Page Out
Page In
Page Out

happens constantly.

System spends more time moving memory than executing work.

Page Fault - occurs when a process accesses a virtual page that is not currently available in RAM.

Process:

Access Virtual Page
       ↓
Page Missing
       ↓
Page Fault
       ↓
OS Loads Page
       ↓
Execution Continues

Page faults are normal.

Examples:

  • accessing swapped memory
  • accessing memory-mapped files
  • first access after allocation

Types:

Minor Page Fault

Page already exists in memory.
Only page table update needed.

Major Page Fault

Page must be loaded from disk.

Major page faults are expensive.

Symptoms:

  • application pauses
  • high disk activity
  • poor performance

Segmentation Fault (Seg Fault) - process attempts to access memory it is not allowed to access.

int* ptr = NULL;
*ptr = 5;

or:
Read invalid address
Write invalid address
Execute invalid address

Operating System response: SIGSEGV

Process terminates.

Common causes:

  • null pointer dereference
  • buffer overflow
  • use-after-free
  • invalid memory access

Python rarely produces seg faults because memory is managed.

Seg faults are more common in:

  • C
  • C++
  • Rust (unsafe code)
  • native extensions

Python Runtime Concepts

Python GIL (Global Interpreter Lock)

A lock inside CPython that allows only one thread to execute Python bytecode at a time.

Conceptually:

Thread A
Thread B
Thread C

      ↓

Only one thread executes Python bytecode
at a given moment

Benefits:

  • simplifies memory management
  • simplifies reference counting
  • reduces interpreter complexity

Tradeoffs:

  • limits CPU-bound thread parallelism
  • multiple threads cannot fully utilize multiple CPU cores

Example:

CPU-heavy workload:

for i in range(1_000_000_000):
    ...

Multiple threads often do not speed this up significantly.

However:

I/O-bound workloads

still benefit greatly:

  • HTTP requests
  • Database calls
  • File operations
  • Network I/O

because the GIL is released while waiting.

Rule of thumb:
CPU-bound → multiprocessing
I/O-bound → threading or async

CPython - the standard and most widely used implementation of Python.

Features:

  • written primarily in C
  • uses reference counting
  • includes cyclic garbage collector
  • contains the GIL

Most Python behavior people discuss refers specifically to CPython.

Protocol

Protocol asks: Does this object have the methods I need?
rather than: Does this object inherit from my base class?

Structural typing.

class Repository(Protocol):
