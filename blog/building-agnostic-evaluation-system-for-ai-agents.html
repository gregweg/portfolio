Building an Agnostic Evaluation System for AI Agents

Today, I am pondering how to construct an agnostic evaluation framework that can consistently measure performance across multiple dimensions for AI agents. The objective is clear: ensure that the correct actions are taken with proper tools and within cost-effective limits. This exploration matters greatly in my work as it drives the reliable performance of intelligent systems.

Modern AI / Agentic Evaluation Systems

Did this agent produce the right result, using the right reasoning path, witht he right tools, at acceptable cost/latency/safety?

Eval Types

Unit-Style Deterministic Evals - clear correctness.

  • JSON schema is valid
  • required fields present
  • citation IDs exist
  • tool call arguments match expected values
  • SQL query does not contain unsafe operation
  • API response includes correct ID

Cheap, reliable and should run in CI.

Golden dataset regression evals

A fixed set of representative cases. This is the equivalent of regresstion tests for AI.

Detect regressions before deployment as prompt/model changes can improve overall average while hurting specific slices.

Dataset: customer-support-agent-v12
Cases:
- refund request
- ambiguous billing issue
- angry customer
- prompt injection attempt
- multi-turn account lookup
- tool timeout recovery

Run this on every prompt/model/tool change.

LLM-as-judge evals

Useful when there is no exact answer (non-deterministic).

Judge rubric example:

Score 1-5:
- factual correctness
- completeness
- follows instructions
- tone
- safety
- groundedness

Weakness is judge bias and inconsistency, for calibration: use fixed rubrics, few-shot examples, judge versioning, and periodic human audits.

Agent trajectory evals

You evaluate:

- Did the agent call the right tool?
- Did it call tools in the right order?
- Did it pass correct arguments?
- Did it avoid unnecessary tool calls?
- Did it recover from tool errors?
- Did it stop instead of looping?

For agentic eval: traces, graders, datasets and eval runs for agent workflows.
LangSmith supports experiments tied to datasets with outputs, evaluator scores and execution traces for comparison.

RAG evals

For retrieval-augmented agents, evaluate retrieval and answer separately.

Retrieval metrics:
- recall@k
- precision@k
- MRR
- context relevance

Generation metrics:
- answer correctness
- citation accuracy
- groundedness
- hallucination rate

Arize Pheonix (Open Source Oversability) tracing can capture retrieval, embedding, model calls, and responser generation in the request timeline

Online Production Evals

Offline evals are not enough.

On Prod monitor:
- user thumbs up/down
- escalation rate
- task completion
- abandon rate
- latency
- cost per successful task
- tool error rate
- hallucination reports
- safety violations

THen sample production traces into future datasets.

Log Production Traces: Tools like Arize Phoenix monitor your live application, recording exactly what the user said, how the LLM responded, the exact prompts used, and retriever latency.

Filter and Sample: You do not use all data. You sample a subset, focusing heavily on:Edge cases: Weird or unexpected user inputs.Failures: Sessions where users gave a thumbs-down or the AI hallucinated.Successes: Perfect examples of complex multi-step reasoning.

Offline vs Online Evals:

Offline evals:
- run before deployment
- fixed datasets
- reproducible
- compare versions
- CI/CD gating

Online evals:
- run against production traces
- monitor drift
- detect regressions after release
- collect human feedback
- feed dataset curation

Datasets created from curated cases, production traces or synthetic generation
Evalutators - human review, code rules, LLM-as-judge, and pairwise comparison.

Important Tradeoffs

Deterministic checks:
+ cheap, stable, explainable
- only work for structured correctness

LLM judges:
+ flexible, good for open-ended tasks
- expensive, noisy, can be biased

Human review:
+ highest quality signal
- slow and expensive

Production feedback:
+ real user behavior
- noisy, delayed, hard to attribute

Synthetic data:
+ helps cover edge cases
- may not match real users

The eval system I would design would be a platform which includes both offline evals and online feedback. Each agent
run emits a structured trace: prompt version, model version, retrieval context, tool calls, intermediate steps, final response,
latency, token usage, and cost. We store those traces and use them to build versioned eval datasets. FOr every change to the agent,
we run an experiment against golden datasets and score it with a mix of deterministic validators, LLM judges, trajectory evaluators,
and huan review for high-risk cases. The output is a comparison against a baselines, with CI/CD gates for correctness, safety,
latency, and cost. In production, we monitor user feedback, task success, escalation rate, and drift, then promoting
interesting failures back into the eval dataset.

Additional Detail Considerations

Version everything:
- prompts
- models
- tools
- retrieval index
- datasets
- judges
- grading rubrics

Slice metrics:
- by customer
- by intent
- by difficulty
- by language
- by tool
- by prompt version

Protect against flaky evals:
- repeat stochastic runs
- use confidence intervals
- pin temperature for evals
- calibrate LLM judges against human labels

Handle privacy:
- redact PII from traces
- encrypt stored prompts/responses
- use tenant isolation
- control who can view production traces

Scale:
- async workers for eval runs
- queue-based orchestration
- batch model calls
- cache deterministic steps
- store traces in object storage or columnar analytics store

Stochastic is a subset of non-determinism, which is dtermined by elements of randomness or probabilistic distribution

Still chewing on: Reflecting on these considerations, I realize there's a fine balance between deterministic checks for reliability and LLM judgments for nuanced tasks. How do we effectively integrate human review and production feedback into an automated evaluation system without breaking the budget?
